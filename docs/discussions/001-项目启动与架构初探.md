## 3. 核心问题：为什么 AI 需要记忆系统？

大模型的"无状态"特性带来以下问题：

- 上下文长度有限，无法承载全部历史
- 对话间状态丢失
- 无法从交互中积累知识
- 重复解释背景、偏好、决策

记忆系统的目标是：**在正确的时间，让 AI 拿到最相关的信息**。

## 4. 记忆分层模型（四层架构）

| 层级 | 作用 | 内容示例 | 技术实现 |
|------|------|----------|----------|
| 工作记忆 | 当前任务上下文 | 正在编辑的代码、当前对话 | 内存变量 |
| 短期记忆 | 近期活动回溯 | 今日修改、最近对话摘要 | SQLite / JSON |
| 长期记忆 | 持久知识 | 项目规范、用户偏好、bug 模式 | 向量数据库 + Embedding |
| 程序记忆 | 规则与模板 | 代码风格、审查清单、Prompt 模板 | Markdown / YAML |

## 5. 关键设计决策：语言中立

用户明确提出：记忆系统必须是**语言中立**（Language-Agnostic）的。

这意味着：
- 核心接口和通信协议必须是跨语言的（HTTP REST、gRPC、MCP Server、CLI）
- 存储格式必须是通用的（JSON、SQLite、Protocol Buffers）
- 任何语言编写的 AI Agent（Python、TypeScript、Go、Rust 等）都能平等使用
- 实现语言可以任选，但对外契约不能绑定特定运行时

## 6. 推荐技术栈

- **存储**：SQLite（元数据、时间线）+ LanceDB / sqlite-vec（向量检索），均为文件型、无需独立服务进程
- **嵌入模型**：支持 OpenAI + 本地模型（如通过 Ollama 暴露的通用 HTTP 接口）
- **接口层**：语言无关协议优先
  - **MCP Server**：AI 工具调用的新兴标准
  - **HTTP REST API**：最通用，任何语言都能调用
  - **CLI**：方便脚本集成和本地调试
- **扩展性**：统一 Schema（JSON/Protobuf），后期可接入图数据库

## 7. 核心挑战

记忆系统的难点不在于"存储"，而在于"检索"和"生命周期管理"：

1. **存储什么？** —— 如何判断信息值得长期记住
2. **如何组织？** —— 纯文本、结构化 JSON 还是知识图谱
3. **何时检索？** —— 显式检索 vs 隐式注入
4. **更新与遗忘** —— 偏好会变，旧信息需要版本化或归档
5. **置信度与来源** —— 区分用户明确陈述与 AI 推测

## 8. 主流实现方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| Prompt 内嵌 | 零依赖，效果最好 | 受限于上下文长度 |
| RAG 式检索 | 可扩展，支持海量记忆 | 检索质量不稳定 |
| 结构化知识库 | 适合复杂推理 | 构建成本高 |
| Agent 自主管理 | 最灵活 | 对模型能力要求高 |

## 9. 参考项目

- **MemGPT**：虚拟上下文管理，内存与磁盘换页
- **LangChain Memory**：对话场景的记忆模块
- **Claude Artifacts/Projects**：通过文件实现半持久化
- **Kimi 长上下文**：用超长窗口缓解记忆问题，但成本高

## 10. 关键设计哲学

> **记忆系统的本质不是"存储"，而是"检索"。**

好的记忆系统需要重点关注：
- 索引策略
- 摘要与蒸馏
- 上下文感知检索
